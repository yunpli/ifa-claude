## 3. Dependency graph

```
                    smartmoney.trade_cal
                        ↓ (used by)
                    ALL date math everywhere
                    
TuShare API
    ↓
[T 17:30] smartmoney etl ────→ smartmoney.raw_*
                                       ↓
[T 17:30] ta daily-etl ────→ ta.*  (parallel)
                                       ↓
[T 17:45] smartmoney compute ────→ factor_daily, sector_state_daily, market_state_daily
                                       ↓                 ↓
[T 18:00] smartmoney evening    ←  reads        market evening (later)
[T 18:00] ta evening                                     ↓
[T 18:30] market evening (reads three-aux + smartmoney sections from DB)
[T 19:00] macro evening
[T 19:00] asset evening
[T 19:00] tech evening
[T 19:30] ningbo evening

[T+1 09:25] premarket_warm_cache.py ────→ var/cache/stk_limit_T+1.csv
[T+1 11:35] market noon (uses rt_k + cache)
[T+1 18:30] market evening (reads T+1 morning + noon hyps from DB)
```

**Critical dependencies**:
1. `smartmoney etl` MUST run before `smartmoney compute` (compute reads raw_*)
2. `smartmoney compute` SHOULD run before any evening report that reads factor_daily / sector_state_daily
3. `macro/asset/tech evening` MUST run before `market evening §06 三辅汇总` reads them via DB
4. `premarket_warm_cache.py` SHOULD run before `market noon` (otherwise noon falls back to live API call, ~1.5s slower)
5. `ta daily-etl` MUST run before `ta evening` (event_signal_daily / blacklist_daily / etc populated)

## 4. Status code interpretation (V2.2 NEW)

`model_outputs.status` column meanings (post-V2.2 schema retry layer):

| Status code | What happened | Operator action |
|---|---|---|
| `parsed` | LLM got it right first time. ✅ | None |
| `fallback_used` | LLM was missing required fields after schema retries; partial output used | Investigate prompt drift (rare) |
| `parse_failed` | LLM returned malformed JSON, both attempts failed | Check LLM provider status |
| `error` | LLM call itself errored (timeout / 5xx / connection) | Check LLM endpoint health |

The fine-grained retry trace (`schema_retry_ok_attempt2`, `schema_retry_partial_after3`) appears in **stdout logs** only:

```
[run a3f9b1c2] starting Market evening report for 2026-04-30 user=default
[freshness] ✓ all fresh
building E1 headline...
  E1 headline done in 4.2s (status=schema_retry_ok_attempt1)
```

If `attempt2` or `attempt3` appears frequently → LLM model is degrading; consider switching `IFA_LLM_PRIMARY_MODEL`.

## 5. Banner staleness warning interpretation

When the report banner shows:

> ⚠ 部分数据未更新至 2026-05-06，最新可用日期为 2026-04-30。

**Operator action**:
1. Check `[freshness] ⚠` log lines from the report run
2. Identify which raw table is behind:
   ```sql
   SELECT 'raw_daily' AS tbl, MAX(trade_date) FROM smartmoney.raw_daily
   UNION ALL SELECT 'raw_moneyflow', MAX(trade_date) FROM smartmoney.raw_moneyflow
   UNION ALL SELECT 'raw_index_daily', MAX(trade_date) FROM smartmoney.raw_index_daily
   UNION ALL SELECT 'raw_sw_daily', MAX(trade_date) FROM smartmoney.raw_sw_daily;
   ```
3. Re-run that day's smartmoney ETL:
   ```bash
   ifa smartmoney etl --report-date 2026-05-06
   ```
4. Re-generate the report.

If TuShare itself doesn't have today's data yet (called too early), wait — `[freshness]` warning is correct but not actionable.

## 6. Emergency procedures

### 6.1 Failed report — partial output exists

```bash
ls ~/claude/ifaenv/out/<mode>/<date>/
# Check most recent HTML — if 0 bytes or truncated, regenerate
ifa generate <fam> --slot <slot> --report-date <date> --user default
```

### 6.2 LLM endpoint down

Switch to fallback in env:
```bash
export IFA_LLM_PRIMARY_BASE_URL=$IFA_LLM_FALLBACK_BASE_URL
export IFA_LLM_PRIMARY_API_KEY=$IFA_LLM_FALLBACK_API_KEY
export IFA_LLM_PRIMARY_MODEL=$IFA_LLM_FALLBACK_MODEL
ifa generate market --slot evening --report-date 2026-04-30
```

### 6.3 TuShare API rate limited

```bash
# Wait 60s + retry
sleep 60
ifa smartmoney etl --report-date 2026-04-30
```

For sustained issues, contact TuShare or rotate token.

### 6.4 DB CHECK constraint violation (legacy V2.1 status codes)

V2.2 maps schema_retry_* statuses to allowed values. If you see a `CheckViolation`:
- Most likely: a custom prompt module bypassed `_persist_model_output`
- Fix: ensure all model_outputs writes go through `_persist_model_output`

### 6.5 Total reset (development only)

```bash
# DROP everything and re-migrate
ifa healthcheck
psql -h localhost -p 55432 -U ifa -d ifavr_test -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public; DROP SCHEMA smartmoney CASCADE; CREATE SCHEMA smartmoney; DROP SCHEMA research CASCADE; CREATE SCHEMA research; DROP SCHEMA ta CASCADE; CREATE SCHEMA ta; DROP SCHEMA stock CASCADE; CREATE SCHEMA stock;"
uv run alembic upgrade head
ifa smartmoney backfill --start 20260101 --end 20260430
```

⚠️ NEVER run on `ifavr` (production database).

## 7. Recommended monitoring queries

### 7.1 Daily freshness check

```sql
SELECT
  'raw_daily' AS tbl, MAX(trade_date) AS latest, NOW()::DATE - MAX(trade_date) AS lag_days
FROM smartmoney.raw_daily
UNION ALL
SELECT 'raw_moneyflow', MAX(trade_date), NOW()::DATE - MAX(trade_date) FROM smartmoney.raw_moneyflow
UNION ALL
SELECT 'raw_sw_daily', MAX(trade_date), NOW()::DATE - MAX(trade_date) FROM smartmoney.raw_sw_daily
UNION ALL
SELECT 'factor_daily', MAX(trade_date), NOW()::DATE - MAX(trade_date) FROM smartmoney.factor_daily;
```

Alert if `lag_days > 1` on a trading day (use `smartmoney.trade_cal` to filter weekends).

### 7.2 LLM cost tracking

```sql
SELECT
  DATE_TRUNC('day', created_at)::DATE AS day,
  COUNT(*) AS llm_calls,
  SUM(prompt_tokens) AS prompt_tokens,
  SUM(completion_tokens) AS completion_tokens,
  AVG(latency_seconds) AS avg_latency_s
FROM report_model_outputs
WHERE created_at >= NOW() - INTERVAL '7 days'
GROUP BY 1
ORDER BY 1 DESC;
```

### 7.3 Schema-retry rate (LLM compliance)

```sql
SELECT
  DATE_TRUNC('day', created_at)::DATE AS day,
  COUNT(*) FILTER (WHERE status = 'parsed') AS first_try_ok,
  COUNT(*) FILTER (WHERE status = 'fallback_used') AS fallback_used,
  COUNT(*) FILTER (WHERE status = 'parse_failed') AS parse_failed,
  COUNT(*) FILTER (WHERE status = 'error') AS errors
FROM report_model_outputs
WHERE created_at >= NOW() - INTERVAL '7 days'
GROUP BY 1
ORDER BY 1 DESC;
```

If `fallback_used` rate climbs > 15%, the LLM is hitting schema-retry; consider re-tuning prompts or switching model.

## 9. Troubleshooting cookbook

### "Report generated but §08 是空的（晚报中报判断 Review）"
- **Cause**: noon report's `_build_n11_review_hooks` didn't `insert_judgment` to DB.
- **Fix**: V2.2 fixes this. Re-run noon report:
  ```bash
  ifa generate market --slot noon --report-date 2026-04-30
  ifa generate market --slot evening --report-date 2026-04-30
  ```

### "中报顶部 banner 报警 'raw_moneyflow latest=4/30，期望 5/6'"
- **Cause**: V2.1 era pre-flight expected today's EOD; V2.2 fixed to expect prev_trading_day.
- **Fix**: pull V2.2 code; re-run noon report.

### "LLM 重试很多次（log 显示 schema_retry_ok_attempt2 频繁）"
- **Cause**: LLM model degrading or prompt too complex.
- **Action**: switch model:
  ```bash
  export IFA_LLM_PRIMARY_MODEL=gpt-5.5
  ```

### "Print PDF 中 §10 scenarios 卡片切到下一页只剩一半"
- **Cause**: page-break-inside not avoiding correctly on wide grids.
- **Fix**: in `styles.css` ensure `.ifa-scenario-card { page-break-inside: avoid; }`.

### "Stock Edge decision_score 看起来不靠谱"
- **Cause**: V2.2 ships with frozen but suboptimal YAML; V2.2.1 will land T3.2/T3.3.
- **Action**: 不要单独依赖 decision_score；与 SmartMoney + TA 信号交叉验证。

### "新机器迁移后第一次跑 macro morning 卡在 'fetching liquidity_snapshot'"
- **Cause**: TuShare token 没生效，或 .env 文件没创建。
- **Action**:
  ```bash
  ifa healthcheck   # checks LLM + TuShare + DB connectivity
  ```

## 11. Glossary of terms (operator-facing)

- **Slot** — 报告时间段：morning (早盘 ~08:00) / noon (中盘 ~11:35) / evening (晚盘 ~18:30)
- **Run mode** — `test` / `manual` / `production`：见 [`docs/run-modes.md`](run-modes.md)
- **Trade calendar** — `smartmoney.trade_cal` 是单一事实源（带调休 + 节假日 + 临时停市）
- **Self-fuse** — 当数据缺失时，section 整节消失（不渲染空架子），见 banner 警告
- **Schema retry** — LLM 漏字段时自动 3 次重试（2/4/8s backoff）
- **Top3** — §01 顶部 3 条**前瞻盘中动作**，每条 ≤22 字
- **Banner staleness warning** — 红框 alert 当 `trade_date < report_date`
- **Pre-flight freshness check** — 报告开工前检查 raw_* 表是否落后
