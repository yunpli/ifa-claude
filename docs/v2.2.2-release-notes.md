> **Release type**: feature release + schema migration
> **Main deliverable**: Smart Money Enhanced (SME) MVP1
> **Upgrade path**: `git pull && git checkout v2.2.2 && uv run alembic upgrade head`

---

## 1. Executive Summary

V2.2.2 ships **Smart Money Enhanced (SME)** as a new production-grade family for A-share资金结构 analysis.

SME is not a refactor of old SmartMoney. It is an independent family:

- Code dependency on old SmartMoney: **zero**.
- Local data dependency: **read-only** access to existing `smartmoney.*` tables.
- New derived storage: all SME outputs live in `sme.*`.
- Customer output: conclusion-first brief, simple language, data-supported inflow/outflow evidence, no algorithm exposition.

The release makes SME ready for daily production, parameter tuning, backtest, OOS, and OOC workflows.

| 22:40 | `scripts/sme_incremental_2240.sh` | same-trading-day incremental ETL |

## 3. Data and Logic Contract

SME uses trading days, not calendar days, for:

- previous day
- next day
- recent N days
- forward 1/3/5/10/20 day labels

Implementation must use canonical trading calendars / trading-date row numbers. Do not use `date +/- N` for trading windows.

All money columns ending in `_yuan` are stored in yuan. Missing market data must remain NULL and degraded; do not silently coerce missing values to zero.

The mandatory developer reference is [`docs/sme-data-logic-contracts.md`](sme-data-logic-contracts.md). Any future SME unit, PIT, feature, state, label, ETL, or report-semantics change must update that document or an adjacent code comment.

## 4. Tuning Readiness

V2.2.2 connects persistent market-structure snapshots to forward labels:

```text
sme_market_structure_daily
        ↓
sme_strategy_eval_daily
        ↓
tuning-ready / bucket-review / profile promotion
```

Current YTD readout is intentionally treated as a tuning signal, not as final truth:

- `secondary`, `crowding_risk`, and `avoid` buckets are more useful than current `primary`.
- Next tuning work should improve bucket ranking and thresholds before adding more narrative complexity.

Parameter policy:

- Continuous parameters are preferred for thresholds, weights, penalties, and flow-intensity surfaces.
- Discrete parameters are reserved for structural choices only.
- Good tuning results are promoted to YAML through `ifa sme tune promote-profile`; promotion checks sample readiness and evaluated profile coverage before writing `active_profile`.
